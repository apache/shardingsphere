# Feature Specification: MCP AI-Native Perfect 100

**Feature Branch**: `001-shardingsphere-mcp`
**Created**: 2026-05-08
**Status**: Ready for Implementation
**Input**: User asks to design ShardingSphere MCP to the state where repeated AI-native usability review answers `100`.

## Goal

Define the smallest complete design package that makes `shardingsphere-mcp` reach a strict `100/100` for native, convenient, comfortable, and clear large-model usage.

`100/100` means:

- Within the supported MCP protocol surface and current ShardingSphere MCP product scope, no known repo-visible optimization space remains after this specification's acceptance checklist passes.
- The score is evidence-based, not sentiment-based. It depends on explicit contracts, deterministic tests, and a maintained scorecard.
- The next time the same review question is asked, the answer can be `100` only when every required gate in this Speckit package is satisfied.

## Scope

This feature is a design and requirement-management package for the MCP subchain. It covers:

- Tool/resource discovery clarity.
- Argument completion and workflow planning comfort.
- Metadata browsing and query continuation.
- Error and empty-state recovery.
- Runtime/readiness visibility.
- Secret-safe diagnostics.
- Deterministic verification that prevents known AI-usability regressions.

## Non-Goals

The following are excluded from the `100/100` promise to avoid over-design:

- Building an autonomous planner or agent runtime.
- Adding vector search, semantic indexing, or cross-session memory.
- Introducing RBAC, approval-token systems, or production change orchestration.
- Running live LLM evaluations as a required CI gate.
- Covering every possible third-party MCP client behavior.
- Adding broad new database-management features unrelated to AI-native MCP usability.

## User Scenarios

### Scenario 1: Model discovers what it can do without guessing

An LLM opens the MCP prompt/resource/tool catalog and can identify every supported action, required argument, safe follow-up, and expected response mode without reading Java code.

**Acceptance**:

- Every public descriptor exposes a consistent `response_mode`.
- Every follow-up uses a predictable `next_actions` schema.
- Missing arguments direct the model to a concrete completion resource or tool.

### Scenario 2: Model continues large metadata exploration safely

An LLM lists databases, schemas, tables, columns, indexes, or rules. If a result is capped, it knows whether to paginate, search, narrow scope, or read a parent resource.

**Acceptance**:

- Capped direct metadata lists expose unambiguous continuation semantics.
- If pagination is not supported, the payload says so and routes the model to `search_metadata`.
- `has_more` never implies a non-existent page.

### Scenario 3: Model recovers from not-found and empty states

An LLM asks for an object that does not exist, is out of scope, or is misspelled. The server responds with structured recovery guidance instead of a dead end.

**Acceptance**:

- Not-found responses include a parent resource when known.
- Search retry arguments are derived from the requested object name.
- The server never invents a URI for an object it has not resolved.

### Scenario 4: Model plans workflow arguments without hidden assumptions

An LLM asks for a safe workflow plan. The response shows which arguments were user-provided, inferred, defaulted, generated by the server, or redacted.

**Acceptance**:

- Workflow plans include argument provenance.
- Single-schema auto-fill is deterministic and visible.
- Ambiguous schema/table/database choices result in a structured question, not a guess.

### Scenario 5: Operator and model verify readiness without leaking secrets

An LLM or operator can check whether MCP is ready and which runtime slices are visible without exposing credentials or production-sensitive values.

**Acceptance**:

- Runtime/readiness resources are secret-free.
- Redaction markers are consistent.
- Health and readiness are usable before deeper workflow actions.

## Score Model

The final score is not an average. It is a gate:

- `100`: all required gates pass, no known in-scope optimization remains, and the scorecard documents the evidence.
- `<100`: any required gate is missing, ambiguous, flaky, or undocumented.

Required scorecard categories:

- Discovery contract.
- Invocation contract.
- Continuation contract.
- Recovery contract.
- Runtime/readiness contract.
- Workflow comfort contract.
- Documentation and packaging contract.
- Deterministic verification contract.

## Priority Split

### P0: Zero-Guessing Contract

P0 removes every known place where an LLM may need to infer server intent.

- Standardize `response_mode` across public tool, resource, prompt, and error responses.
- Require `order`, `requires_user_approval`, and clear action type on every `next_actions` entry.
- Add dependency information only when the follow-up sequence actually depends on prior steps.
- Fix capped metadata-list semantics so continuation is either real pagination or explicit search/narrowing guidance.
- Make completion recovery nearest-resource aware for missing `database`, `schema`, `table`, `column`, and `index`.
- Add deterministic single-schema auto-fill where exactly one visible schema exists.
- Add not-found and empty-state recovery with parent resources and retry-ready search arguments.
- Add a contract test suite that fails when any known `100` gap reappears.

### P1: Comfortable Native Use

P1 makes the MCP feel natural for model use without expanding the feature scope.

- Add secret-free readiness and health contracts.
- Add runtime visibility resource data for per-database metadata, capability, feature, and driver readiness.
- Add workflow argument provenance.
- Standardize redaction markers and redaction summaries.
- Add a small domain lexicon for common Chinese data-governance intents.
- Align prompts, tool descriptions, and capability descriptors on the same terminology.
- Add deterministic local MCP client smoke coverage without requiring a live model.

### P2: Proof and Polish

P2 makes `100` durable and reviewable.

- Add optional request or correlation id in recovery-safe responses.
- Add packaging metadata hints for MCP server discovery where the current distribution already supports them.
- Add descriptor linting for unknown navigation types and schema drift.
- Add a generated or maintained `100` scorecard artifact that lists evidence and remaining known non-goals.

## Out of Scope Boundaries

If a future reviewer finds an improvement that belongs to a non-goal, it does not reduce this feature's `100` score.
Record it as a future product direction, not as an in-scope defect.

If a future reviewer finds an in-scope ambiguity not covered by these requirements, the score immediately drops below `100` until the requirement, implementation, and verification are updated.
