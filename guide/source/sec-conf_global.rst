Конфигурация ``global.yaml``
=============================

Файл ``global.yaml`` — это конфигурационный файл Apache ShardingSphere-Proxy, 
в котором задаются системные, общие и инфраструктурные настройки прокси-сервера.

Он не зависит от конкретных логических баз данных, а определяет поведение всего экземпляра ShardingSphere-Proxy.

В этом файле настраиваются следующие компоненты и параметры:

- режим работы (``mode``) — одиночный или кластерный;
- аутентификация и авторизация (``authority``);
- глобальные системные свойства (``props``);
- распределённые транзакции (``transaction``);
- SQL-парсер (``sqlParser``);
- движок федеративного выполнения запросов (``sqlFederation``);
- система логирования (``logging``). 

.. compound::

  Файл ``global.yaml`` располагается по пути:

  .. code-block:: properties

    %SHARDINGSPHERE_PROXY_HOME%/conf/global.yaml

  где ``%SHARDINGSPHERE_PROXY_HOME%`` — это корневая директория установки ShardingSphere-Proxy.

Настройка режима
----------------------

Apache ShardingSphere предоставляет два режима работы: автономный режим и режим кластера.

Standalone mode (Автономный режим)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

*Автономный режим (Standalone)* — это базовый режим работы ShardingSphere-Proxy, в котором каждый 
экземпляр прокси функционирует изолированно и не взаимодействует с другими узлами.  

В этом режиме:  

- все метаданные (информация о шардинге, пользователях, правилах, источниках данных и т.д.) 
  хранятся локально для каждого экземпляра Proxy;  
- любые изменения конфигурации (например, добавление новой таблицы или обновление правил маршрутизации) 
  необходимо вносить вручную в каждый экземпляр, иначе возникнет несогласованность данных между Proxy;  
- экземпляры Proxy не обмениваются информацией о состоянии друг друга;  
- синхронизация метаданных между несколькими экземплярами отсутствует.  

Автономный режим рекомендуется использовать только для локальной разработки, тестирования и отладки.  
Для промышленной эксплуатации рекомендуется *кластерный режим (Cluster mode)*.  


Режим работы указывается в файле конфигурации ``global.yaml``:

.. code-block:: yaml

    mode:
      type: Standalone  # тип режима работы: Standalone (по умолчанию) или Cluster
      repository:       # настройка хранилища метаданных (Persist Repository)
        type:           # тип хранилища метаданных
        props:          # свойства хранилища метаданных
          <ключ>: <значение>
          <ключ>: <значение>
          ...


ShardingSphere использует специальное хранилище метаданных (Persist Repository) для сохранения и загрузки конфигурации.
В режиме ``Standalone`` это хранилище является локальным. По умолчанию для хранения используется встроенная база данных ``H2``, 
работающая через JDBC.

.. rubric:: Пример конфигурации
  :heading-level: 4

.. code-block:: yaml

    mode:
      type: Standalone
      repository:
        type: JDBC

Таким образом, все настройки и метаданные сохраняются локально, и при запуске Proxy они автоматически подгружаются из этого хранилища.  


Cluster mode (Кластерный режим)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

*Кластерный режим (Cluster)* предназначен для работы нескольких экземпляров ShardingSphere-Proxy как единой распределённой системы.  
В этом режиме все узлы Proxy синхронизируют между собой метаданные и состояние через централизованный *реестр конфигураций (Registry Center)*.

Реестр служит хранилищем и координатором для всех Proxy-экземпляров в кластере.  

Он обеспечивает:

- централизованное хранение метаданных (правила шардинга, источники данных, пользователи, схемы и т.д.);  
- регистрацию и управление всеми активными узлами Proxy;  
- рассылку уведомлений об изменениях конфигурации;  
- мониторинг и управление состоянием экземпляров Proxy (например, при отказах или перезапусках).  

Таким образом:

- изменения конфигурации на одном узле автоматически синхронизируются со всеми остальными;  
- кластер продолжает работу при отказе отдельных узлов (обеспечивается высокая доступность, HA);  
- новые Proxy-инстансы можно подключать динамически, без ручной настройки.  

В качестве Registry Center обычно применяются:  

- ``ZooKeeper``
- ``Etcd``
- ``Consul``

Режим работы задаётся в конфигурационном файле ``global.yaml``:  

.. code-block:: yaml

    mode:
      type: Cluster          # режим работы: Standalone (по умолчанию) или Cluster
      repository:            # хранилище метаданных (persist repository)
        type:                # тип хранилища (zookeeper / etcd / consul)
        props:               # свойства подключения к реестру
          namespace:         # пространство имён для хранения данных ShardingSphere в реестре
          server-lists:      # список серверов реестра (host:port, через запятую)
          <ключ>: <значение> # дополнительные параметры 
          <ключ>: <значение>
          ...

Для работы с конкретным типом хранилища необходимо добавить соответствующую зависимость ``Maven``.  
Например, для ``ZooKeeper``:

.. code-block:: xml

    <dependency>
        <groupId>org.apache.shardingsphere</groupId>
        <artifactId>shardingsphere-cluster-mode-repository-zookeeper</artifactId>
        <version>${shardingsphere.version}</version>
    </dependency>

.. rubric:: Пример конфигурации
  :heading-level: 4

.. code-block:: yaml

    mode:
    type: Cluster
    repository:
        type: ZooKeeper
        props: 
           namespace: governance
           server-lists: localhost:2181
           retryIntervalMilliseconds: 500
           timeToLiveSeconds: 60


Настройка аутентификации и авторизации
--------------------------------------------

При подключении к ShardingSphere-Proxy пользователи проходят процедуру аутентификации и авторизации.

Все пользователи, имеющие доступ к Proxy, задаются в конфигурационном файле ``global.yaml``.

.. code-block:: yaml

  authority:
    users:
      - user:       # логин пользователя в формате имя@хост
        password:   # пароль пользователя
        admin:      # true=администратор, имеет доступ ко всем БД (по умолчанию false) 
        authenticationMethodName: # имя метода аутентификации
    authenticators:         # список доступных методов аутентификации
    
        type:               # провайдер привилегий: ALL_PERMITTED (по умолчанию) или 
                            # DATABASE_PERMITTED

Пользователи (``users``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

- Каждый пользователь задаётся в формате: ``имя@хост``:

    - ``root@%`` — доступ с любого хоста (% или пустая строка).
    - ``user@127.0.0.1`` — доступ только с локального хоста.

- Имя пользователя и хоста нечувствительны к регистру.
- Пароль является обязательным.л

Методы аутентификации (``authenticators``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

ShardingSphere поддерживает несколько алгоритмов аутентификации.
Они указываются в параметре ``type`` и могут быть заданы индивидуально для каждого пользователя.

Поддерживаемые значения:

    - ``SRP``    
    - ``SRP224``
    - ``SRP256`` (по умолчанию) 
    - ``SRP384``   
    - ``SRP512``    
    - ``LEGACY_AUTH``

Если параметр ``authenticationMethodName`` не указан, ShardingSphere автоматически выбирает метод,
соответствующий протоколу клиента. По умолчанию используется ``SRP256``.

Провайдеры привилегий (``privilege``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Провайдер привилегий определяет, какие логические базы данных доступны пользователю.
Доступны два режима:

- ``ALL_PERMITTED``

Каждый пользователь имеет полный доступ ко всем базам данных. Используется по умолчанию.
Не рекомендуется для использования в продакшене (будет удалён в будущих версиях).

- ``DATABASE_PERMITTED``

Пользователям предоставляется доступ только к конкретным логическим базам данных. 
Настройка выполняется с помощью параметра ``user-database-mappings``.
Рекомендуется использовать в продакшене.

Примеры конфигураций
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

.. rubric:: Минимальная конфигурация
  :heading-level: 4

.. compound::

  .. code-block:: yaml

      authority:
        users:
          - user: root@%
            password: root
          - user: sharding
            password: sharding


  *Пояснения:*

  - определены два пользователя: ``root@%`` и ``sharding``;
  - ``authenticationMethodName`` не указан. Это значит используется ``SRP256``;
  - провайдер привилегий не указан, значит будет использоваться ``ALL_PERMITTED``.

.. rubric:: Указание метода аутентификации
  :heading-level: 4

.. compound::

  .. code-block:: yaml

      authority:
        users:
          - user: root@127.0.0.1
            password: root
          - user: sharding
            password: sharding
            authenticationMethodName: legacy
        authenticators:
          legacy:
            type: SRP224
        privilege:
          type: ALL_PERMITTED

  *Пояснения:*

  - определены два пользователя: ``root@127.0.0.1`` и ``sharding``;
  - используется метод аутентификации ``SRP224`` для пользователя ``sharding``;
  - для пользователя ``root@127.0.0.1`` не указан метод аутентификации, значит будет использоваться SRP256;
  - указан провайдер привилегий ``ALL_PERMITTED``.

.. rubric:: Ограниченные привилегии (``DATABASE_PERMITTED``)
  :heading-level: 4

.. compound::

  .. code-block:: yaml

      authority:
        users:
          - user: root@127.0.0.1
            password: root
            admin: true
          - user: sharding
            password: sharding
          - user: test
            password: sharding
        privilege:
          type: DATABASE_PERMITTED
          props:
            user-database-mappings: sharding@%=*, test@%=test_db, test@%=sharding_db

  *Пояснения:*

  - определены три пользователя: ``root@127.0.0.1`` (администратор), ``sharding`` и ``test``.
  - ``authenticators`` и ``authenticationMethodName`` не определены. Значит используется ``SRP256``;
  - Указан провайдер привилегий ``DATABASE_PERMITTED``:

    - пользователь ``sharding`` имеет доступ ко всем логическим базам ``(*)``;
    - пользователь ``test`` имеет доступ только к ``test_db`` и ``sharding_db``.

Системные свойства ShardingSphere-Proxy (``props``)
---------------------------------------------------------

Apache ShardingSphere предоставляет множество свойств конфигурации системы, которые пользователи могут настраивать через ``global.yaml``.

.. tabularcolumns:: |>{\ttfamily\arraybackslash}\X{4}{16}|>{\ttfamily\arraybackslash}\X{2}{16}|>{\ttfamily\arraybackslash}\X{2}{16}|>{\arraybackslash}\X{8}{16}|
.. list-table:: Список свойств
   :class: longtable
   :name: table:prop
   :header-rows: 1

   * - Название
     - Тип данных
     - Значение по умолчанию
     - Описание
   * - system-log-level
     - String
     - INFO
     - Уровень логирования Proxy (``TRACE, DEBUG, INFO, WARN, ERROR``).
   * - max-connections-size-
       per-query
     - int
     - 1
     - Максимальным количеством подключений к физическим базам данных, которые могут использоваться для выполнения одного SQL-запроса.
   * - kernel-executor-size 
     - int
     - infinite
     - Размер пула потоков ядра. Каждый источник данных ShardingSphere использует независимый пул потоков, 
       и разные источники данных на одной JVM не используют общие пулы потоков.
   * - proxy-frontend-flush-
       threshold
     - int
     - 128
     - Порог количества строк, накопленных в буфере перед отправкой клиенту.
   * - sql-show
     - boolean
     - false
     - Выводить ли SQL-запросы в логах.
   * - check-table-metadata-
       enabled
     - boolean
     - false
     - Если ``true``, при каждом SQL-запросе Proxy проверяет:

       - Существуют ли указанные таблицы в метаданных.
       - Соответствуют ли имена таблиц и структура тем, что указаны в конфигурации.

       Если ``false``, Proxy пропускает проверку, полагаясь на метаданные, загруженные при старте или через persist repository.
   * - proxy-backend-query-
       fetch-size
     - int
     - -1
     - Управляет размером выборки данных (``fetch size``) при запросах к физическим базам данных, т.е. 
       сколько строк Proxy извлекает за один раз из шарда при выполнении ``SELECT``-запроса.

       -1 означает, что ShardingSphere-Proxy будет использовать минимальное значение ``fetch size``, рекомендуемое 
       JDBC-драйвером для конкретной базы данных.
   * - proxy-frontend-
       executor-size
     - int
     - 0 
     - Размер пула потоков Netty для обработки клиентских запросов.

       0 означает, что ShardingSphere-Proxy позволяет Netty самому управлять пулом потоков.
   * - proxy-frontend-max-
       connections
     - int
     - 0
     - Максимальное количество одновременных соединений от клиентов к Proxy. 

       0 или отрицательное число - лимит отсутствует, можно подключаться неограниченному числу клиентов.
   * - proxy-default-port
     - String
     - 3307
     - Порт, на котором работает ShardingSphere-Proxy.
   * - proxy-netty-backlog
     - int
     - 1024
     - Очередь соединений Netty.
   * - cdc-server-port 
     - String
     - 33071
     - Порт, на котором работает сервер CDC (Change Data Capture).
   * - proxy-frontend-ssl-
       enabled
     - boolean
     - false
     - Включить SSL для соединений с клиентами.
   * - proxy-frontend-ssl-
       cipher
     - String
     - ''
     - Набор шифров SSL.
   * - proxy-frontend-ssl-
       version
     - String
     - TLSv1.2,
       TLSv1.3
     - Разрешённые версии TLS.

.. rubric:: Пример конфигурации
  :heading-level: 4

.. code-block:: yaml

      props:
        system-log-level: INFO
        max-connections-size-per-query: 1
        proxy-frontend-flush-threshold: 128
        sql-show: false
        check-table-metadata-enabled: false
        proxy-backend-query-fetch-size: -1
        proxy-frontend-executor-size: 0 
        proxy-frontend-max-connections: 0 
        proxy-default-port: 3307 
        proxy-netty-backlog: 1024
        cdc-server-port: 33071 
        proxy-frontend-ssl-enabled: false
        proxy-frontend-ssl-cipher: ''
        proxy-frontend-ssl-version: TLSv1.2,TLSv1.3


Настройка распределенных транзакций (``transaction``)
-------------------------------------------------------------

Модуль ``transaction`` в файле ``global.yaml`` отвечает за конфигурацию распределённых 
транзакций (Distributed Transactions) в ShardingSphere-Proxy.

.. code-block:: yaml

    transaction:
      defaultType:      # Тип транзакций по умолчанию, который будет использоваться Proxy.
      providerType:     # Механизм управления транзакциями

Основные параметры настройки распределённых транзакций:

- ``defaultType``

  Определяет тип транзакций, используемых по умолчанию. Допустимые значения:

  - ``LOCAL`` — локальные транзакции (по умолчанию). 
    Работают в пределах одной базы (шарда), как обычные транзакции JDBC.

  - ``XA`` — полноценные XA-транзакции (двухфазный коммит). На данный момент не поддерживаются.

  - ``BASE`` — «мягкие» транзакции (Best Efforts Delivery). Для СУБД Ред Базы Данных и Firebird не поддерживается.

- ``providerType``

  Задаёт конкретный механизм управления транзакциями. Используется только при выборе ``XA`` или ``BASE``. Поддерживаемые значения:

  - ``Atomikos``
  - ``Narayana``
  - ``Bitronix``

.. rubric:: Примеры
  :heading-level: 4

.. code-block:: yaml

    transaction:
      defaultType: LOCAL     # обычные локальные транзакции

.. code-block:: yaml

    transaction:
      defaultType: XA        # XA-транзакции по умолчанию
      providerType: Atomikos


Настройки SQL-парсера (``sqlParser``)
---------------------------------------------

Модуль ``sqlParser`` в ``global.yaml`` отвечает за конфигурацию встроенного SQL-парсера ShardingSphere-Proxy.

Перед выполнением любого SQL-запроса Proxy выполняет следующие шаги:

1. Парсинг SQL и разбор его синтаксиса.
2. Определение логики запроса: участвующие таблицы, ключи шардинга, условия фильтрации и сортировки.
3. Построение плана выполнения: определение целевых шардов, на которые будет направлен запрос.

Модуль ``sqlParser`` позволяет настроить поведение парсера для балансировки между точностью разбора и производительностью.

.. code-block:: yaml

  sqlParser:
    sqlStatementCache:               # Локальный кэш операторов SQL
        initialCapacity: 2000        # Начальная ёмкость локального кэша
        maximumSize: 65535           # Максимальная емкость локального кэша
    parseTreeCache:                  # Кэш для деревьев разбора SQL
        initialCapacity: 128         # начальная ёмкость кэша синтаксических деревьев
        maximumSize: 1024            # максимальное размер кэшируемых деревьев


Настройки компонента Federation Execution Engine (``sqlFederation``)
----------------------------------------------------------------------------------

Модуль ``sqlFederation`` в ``global.yaml`` отвечает за конфигурацию Federation Execution Engine — экспериментального компонента ShardingSphere, 
обеспечивающего поддержку федеративных запросов.

.. note::

  Эта функция экспериментальная и в настоящее время не подходит для использования в основных производственных средах системы.

Федеративные запросы позволяют выполнять один SQL-запрос одновременно над несколькими источниками данных 
(шардами или даже разными типами баз) с последующим объединением и обработкой результатов на уровне ShardingSphere-Proxy.
Включение этого модуля позволяет Proxy обрабатывать такие федеративные запросы.

.. code-block:: yaml

    sqlFederation:
      sqlFederationEnabled: true   # включение/выключение Federation Engine
      executionPlanCache:
        initialCapacity: 2000      # начальный размер кэша планов выполнения
        maximumSize: 65535         # максимальный размер кэша планов выполнения


Настройка логирования (``logging``)
--------------------------------------------

Модуль ``logging`` в файле ``global.yaml`` управляет тем, как ShardingSphere-Proxy ведёт журналирование SQL-запросов и системных событий.

Параметр sql-show: true в блоке props лишь включает базовый вывод SQL-запросов в стандартный поток, 
тогда как модуль logging предоставляет расширенные возможности настройки:

- Назначение логов: консоль, файл, удалённый лог-сервис через SLF4J/Logback/Log4j.
- Содержание логов

  - только SQL-запросы,
  - SQL с параметрами,
  - результаты выполнения,
  - ошибки парсинга и маршрутизации.

- Уровни логирования: ``INFO, DEBUG, WARN, ERROR``.

- Формат вывода (например, ``JSON`` для последующей обработки).

.. code-block:: yaml

   logging:
     loggers:
     - loggerName: ShardingSphere-SQL
       additivity: true
       level: INFO
       props:
         enable: false
