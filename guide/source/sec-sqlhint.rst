SQL Hint 
=========

В настоящее время большинство реляционных баз данных предоставляют SQL Hint как дополнение к синтаксису SQL. 
SQL Hint позволяет пользователям вмешиваться в процесс выполнения SQL-запросов с помощью встроенного 
синтаксиса подсказок базы данных, чтобы реализовывать специальные функции или оптимизировать выполнение SQL.

ShardingSphere также поддерживает синтаксис SQL Hint, позволяющий пользователям принудительно управлять 
маршрутизацией для шардинга и разделения чтения и записи, а также выполнять прямую передачу запроса в 
указанный источник данных.

Синтаксис SQL Hint в ShardingSphere должен записываться внутри SQL-запроса в виде комментария. 
В настоящее время поддерживается только формат комментариев ``/* */``. 
Содержимое Hint должно начинаться с префикса ``SHARDINGSPHERE_HINT:``, после которого задаются 
пары атрибут–значение для различных функций. Если указано несколько атрибутов, они разделяются запятыми.

Общий формат SQL Hint в ShardingSphere:

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: {key} = {value}, {key} = {value} */ SELECT * FROM t_order;

В SQL Hint ShardingSphere можно задавать различные атрибуты. 
Для совместимости со старыми версиями синтаксиса SQL Hint допускается использование 
алиасов, определённых для тех же параметров:

``SHARDING_DATABASE_VALUE, SHARDING_TABLE_VALUE``
--------------------------------------------------

В SQL Hint ShardingSphere предусмотрены специальные атрибуты, позволяющие принудительно задать 
значения шардирования и тем самым явно управлять маршрутизацией запроса независимо от вычисления шард-ключей на основе условий SQL.
Атрибуты работают только если в конфигурации задана Hint-стратегия шардирования.
Если используется стандартная стратегия, Hint-параметры будут проигнорированы.

``{table}.SHARDING_DATABASE_VALUE`` — используется для задания значения шардирования по источнику данных (базе данных), 
соответствующего логической таблице {table}. При указании нескольких атрибутов они разделяются запятыми.
Если в конфигурации используется только шардирование по базам данных, 
то при принудительной маршрутизации в определённый источник данных можно 
задать значение шардирования через ``SHARDING_DATABASE_VALUE`` без указания имени ``{table}``.

``{table}.SHARDING_TABLE_VALUE`` — используется для задания значения шардирования на уровне таблицы, 
для указанной логической таблицы ``{table}``. Несколько атрибутов также разделяются запятыми.

Пример использования SQL Hint для шардирования:

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: t_order.SHARDING_DATABASE_VALUE=1, t_order.SHARDING_TABLE_VALUE=1 */ SELECT * FROM t_order;


``WRITE_ROUTE_ONLY``
---------------------

Атрибут ``WRITE_ROUTE_ONLY`` используется в SQL Hint для механизма разделения чтения и записи (Read-write Splitting)
и позволяет принудительно направить текущий запрос на источник данных записи.

Если значение параметра установлено в ``true``, запрос направляется только на 
источник данных записа, 
независимо от стандартной стратегии балансировки чтения. 
Это применяется в случаях, когда необходимо гарантировать получение актуальных данных сразу после операций записи либо обойти репликационную задержку.

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: WRITE_ROUTE_ONLY=true */ SELECT * FROM t_order;


``DATA_SOURCE_NAME``
---------------------

Атрибут ``DATA_SOURCE_NAME`` используется в SQL Hint для принудительной отправки SQL-запроса 
напрямую в конкретный физический источник данных. В значении параметра необходимо указать имя 
источника данных, зарегистрированного в логической базе ShardingSphere.

Это механизм прямой маршрутизации запроса к источнику данных через SQL Hint.

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: DATA_SOURCE_NAME=ds_0 */ SELECT * FROM t_order;


``SKIP_SQL_REWRITE``
---------------------

Атрибут ``SKIP_SQL_REWRITE`` используется в SQL Hint для управления фазой переписывания SQL-запроса в ShardingSphere.
Если задать значение ``true``, текущий запрос будет выполнен без прохождения этапа переписывания.

Фаза переписывания — это внутренняя стадия обработки запроса в ShardingSphere, 
на которой он может изменяться (например, подставляются реальные имена таблиц, 
добавляются служебные условия, модифицируются агрегаты и т.п.). Отключение этой стадии позволяет отправить запрос в исходном виде.

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: SKIP_SQL_REWRITE=true */ SELECT * FROM t_order;

В этом случае ShardingSphere не будет выполнять преобразование SQL и передаст его дальше в исходном виде.


Этот параметр обычно применяется для отладки, тестирования или специальных сценариев, 
где необходимо обойти механизм переписывания запроса. В обычной эксплуатации отключать стадию переписывания не рекомендуется, 
поскольку это может нарушить корректную маршрутизацию и выполнение запроса.


``SKIP_METADATA_VALIDATE``
---------------------------

Атрибут ``SKIP_METADATA_VALIDATE`` используется в SQL Hint для управления стадией проверки метаданных запроса.
Если установить значение ``true``, ShardingSphere отключит проверку метаданных SQL-запроса.

Проверка метаданных — это внутренняя стадия обработки запроса, на которой система сверяет 
структуру запроса со схемой базы данных: проверяет существование таблиц, колонок, типов данных и 
корректность ссылок. Пропуск этой проверки позволяет выполнить запрос без предварительной валидации.

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: SKIP_METADATA_VALIDATE=true */ SELECT * FROM t_order;

Опция используется преимущественно в служебных, тестовых или нестандартных сценариях 
(например, при динамически изменяющейся схеме). В продуктивной среде отключение проверки 
метаданных не рекомендуется, так как может привести к выполнению некорректных запросов или ошибкам на уровне реальной БД.

``DISABLE_AUDIT_NAMES``
-------------------------

``DISABLE_AUDIT_NAMES`` — это необязательный атрибут SQL Hint, который позволяет отключать один или несколько SQL-аудиторов 
для конкретного запроса в ShardingSphere.

Если требуется отключить несколько аудиторов, их имена перечисляются через запятую.

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: DISABLE_AUDIT_NAMES=sharding_key_required_auditor */ SELECT * FROM t_order;


``SHADOW``
------------

``SHADOW`` — это необязательный атрибут SQL Hint, предназначенный для работы с теневыми (shadow) источниками данных в ShardingSphere.

Если значение параметра установлено в ``true``, текущий SQL-запрос будет принудительно направлен 
не в основную рабочую базу данных, а в настроенный shadow-источник данных.

.. code-block:: sql

  /* SHARDINGSPHERE_HINT: SHADOW=true */ SELECT * FROM t_order;
