# Detailed Design: Algorithm Recommendation and Property Lifecycle

## 1. 目标

这份文档把“算法推荐”和“密钥/参数在什么时机输入”落成可实现的设计。

核心原则只有一句话：

先确定算法，再采集参数，再生成可执行工件。

## 2. 推荐输入维度

### 2.1 Encrypt 维度

- 是否需要解密展示
- 是否需要等值查询
- 是否需要模糊查询
- 字段语义
- 是否已有现存规则需要修改
- Proxy 当前可见算法池

### 2.2 Mask 维度

- 字段语义
- 期望展示效果
- 是否已有现存规则需要修改或删除
- Proxy 当前可见算法池

## 3. 算法候选池来源

### 3.1 发现层

候选池基于 Proxy 当前可见插件，发现入口包括：

- `SHOW ENCRYPT ALGORITHM PLUGINS`
- `SHOW MASK ALGORITHM PLUGINS`

### 3.2 富化层

仅靠 `show plugins` 不足以做 encrypt 最终推荐，因此需要再补：

- 内置算法的 SPI 元数据
- 当前实例可探测到的能力位
- MCP 维护的属性模板与推荐标签

### 3.3 回退层

如果是自定义 SPI 算法且缺少完整元数据：

- 仍可进入候选池
- 但必须标注为“能力待确认”
- 如用户选择该算法，MCP 必须追加确认，而不是假定其支持某种查询能力

## 4. 推荐决策流程

1. 从自然语言抽取结构化意图
2. 拉取当前 Proxy 的可见算法池
3. 富化 encrypt 能力位
4. 用需求约束筛掉明显不兼容候选
5. 对剩余候选排序并生成推荐理由
6. 让用户选择推荐算法或覆盖为指定算法

## 5. Encrypt 能力匹配规则

### 5.1 解密需求

- `requiresDecrypt=true` 时，不支持解密的算法不能作为首选。

### 5.2 等值查询需求

- `requiresEqualityFilter=true` 时，推荐结果必须满足以下二者之一：
  - 主算法本身支持等值过滤
  - 可搭配 `assisted_query` 方案

### 5.3 模糊查询需求

- `requiresLikeQuery=true` 时，推荐结果必须满足以下二者之一：
  - 主算法本身支持模糊查询
  - 可搭配 `like_query` 方案

## 6. 参数采集采用两段式模型

### 6.1 阶段 A：算法确定

这一阶段只做以下事情：

- 给出推荐列表
- 说明每个候选为什么被推荐或被降级
- 确认最终算法

此时不要求用户先输入密钥。

### 6.2 阶段 B：参数采集

算法确定后，MCP 再根据算法模板收集参数。

参数分四类：

- `required-secret`
- `required-non-secret`
- `optional-secret`
- `optional-non-secret`

只有在必填参数都齐全后，MCP 才能生成最终可执行工件。

## 7. 参数采集时机

### 7.1 必须晚于算法选择

原因：

- 不同算法需要的属性键不一样
- 不同算法的必填项不一样
- 过早采集会增加用户负担并提升泄露风险

### 7.2 必须早于最终工件生成

原因：

- DistSQL / 规则工件需要完整属性
- review 阶段需要展示参数摘要
- manual-only 模式也需要形成可执行工件

因此正确顺序是：

`算法选择 -> 参数采集 -> 工件生成 -> 审阅 -> 执行`

## 8. review 与敏感参数展示策略

### 8.1 默认 review

默认 review 中：

- 非敏感参数可直接展示
- 敏感参数只展示掩码摘要

例如：

- `aes-key-value = ******`
- `iv = ******`

### 8.2 Manual-only 的额外约束

`manual-only` 模式下，用户最终需要可执行工件。

因此设计上要区分两份输出：

- `review-safe preview`
  - 默认展示
  - 敏感值打码
- `executable artifact`
  - 只在用户明确要求查看或导出执行工件时返回
  - 用于用户自行执行

这样可以避免把明文密钥在默认 review 中直接摊开。

## 9. 会话上下文中的敏感参数处理

- 敏感参数只保存在运行态会话上下文
- 不写入审计库
- 不写入长期业务表
- summary 默认只保留掩码后的参数摘要
- 工作流完成或超时后，敏感参数上下文应失效

## 10. 自定义 SPI 算法的参数处理

### 10.1 有模板时

- 直接按模板追问
- 区分必填、可选、敏感三类属性

### 10.2 无模板时

- 提示用户该算法缺少标准属性模板
- 允许用户按 `key=value` 形式手工补充
- 标注该配置缺少自动校验能力

## 11. 生成工件前的最小检查

在进入工件生成阶段前，MCP 必须至少完成以下检查：

- 算法已明确
- 必填参数已齐全
- 加密能力与查询需求不冲突
- 目标逻辑表和逻辑列在 Proxy 逻辑视图中存在
- 执行模式已明确

## 12. 推荐输出规范

推荐结果至少包含：

- `algorithmType`
- `source`
- `recommendationReason`
- `riskNotes`
- `requiredProperties`
- `optionalProperties`
- `secretProperties`

如果某算法不是首选，也应尽量给出原因，例如：

- 不支持解密
- 不支持等值查询
- 模糊查询能力待确认
- 参数模板缺失

## 13. 用户体验要求

- 不要求用户一上来就懂 DistSQL。
- 不要求用户一开始就提供全部属性键。
- 如果自然语言里已经给了算法名称，MCP 仍要校验该算法是否存在于当前 Proxy 可见池中。
- 如果自然语言里已经给了密钥，也应在参数采集阶段重新整理，并在后续展示中打码。
